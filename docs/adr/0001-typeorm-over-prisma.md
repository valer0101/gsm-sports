# 0001 — TypeORM over Prisma

**Status:** Accepted
**Date:** 2025-09 (retroactive — captured in May 2026 doc refresh)

## Context

The API needs an ORM for PostgreSQL. The two mainstream choices for NestJS in 2025 are TypeORM (NestJS's first-party recommendation) and Prisma (more popular in greenfield Node projects).

Constraints:
- Heavy use of JSONB for sport-specific configuration (`SportConfig`, `bracketData`, match results) — schema-flexible columns.
- Need decorator-based entities so DTOs can share type info via `class-validator`.
- Migrations must be reviewable, generatable, and runnable from CI without a separate binary on the runner.
- Team familiarity: NestJS first-party stack reduces onboarding friction.

## Decision

Use **TypeORM** with the `@nestjs/typeorm` integration.

## Consequences

Positive:
- First-party NestJS integration — `TypeOrmModule.forRootAsync`, `@InjectRepository`, transaction helpers, DataSource usage in scripts.
- Decorator entities (`@Entity`, `@Column`, `@OneToMany`) line up with `class-validator` DTOs for end-to-end type safety.
- Migrations are TypeScript files (reviewable in PRs) generated by `typeorm migration:generate` and runnable via `typeorm migration:run`.
- JSONB columns work natively (`@Column('jsonb')`) — no schema regeneration when shape evolves.
- No separate binary needed in CI / production deploy (Prisma's `prisma generate` step is a frequent failure mode in CI matrices).

Negative:
- TypeORM's query builder is more verbose than Prisma's chained `findMany`.
- Documentation quality is uneven — community examples often outdated.
- No equivalent to Prisma's typed query results without manual `Repository` typing.
- Migrations must be reviewed carefully — generated diffs occasionally include unrelated noise from connection options.

Mitigations:
- Codify "use Repository methods or QueryBuilder; never raw SQL with string concatenation" in `CLAUDE.md`.
- In code review, scan generated migrations for unintended changes before merging.
- When TypeORM is wrong for a query (rare, complex aggregations), reach for parameterised raw SQL via `dataSource.query(sql, params)` — never string-concat user input.
